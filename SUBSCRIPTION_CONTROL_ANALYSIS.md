# 🔒 Анализ механизмов контроля подписок

## ✅ Что РАБОТАЕТ после перехода на Shadowsocks:

### 1. **Проверка активности при выдаче конфига через бота** ✅

**Где:** `bot.py` → `myconfig_command()`
```python
is_active = db.check_user_active(user.id)
if not is_active:
    # Деактивирует пользователя
    db.deactivate_user(user.id)
    # Не выдает конфиг
```

**Результат:** Пользователь с истекшей подпиской **НЕ ПОЛУЧИТ** конфиг через `/myconfig`

---

### 2. **Блокировка доступа к конфигу через HTTP** ✅

**Где:** `config_server.py` → `get_config()`
```python
if DB_AVAILABLE:
    is_active = db.check_user_active(user_id)
    if not is_active:
        abort(403)  # Forbidden - не отдает конфиг
```

**Результат:** Прямой доступ к конфигу через URL блокируется (HTTP 403)

---

### 3. **Проверка активности перед покупкой** ✅

**Где:** `bot.py` → `handle_payment_screenshot()`
```python
if existing_user and db.check_user_active(user.id):
    # Не позволяет купить, если уже есть активная подписка
```

**Результат:** Предотвращает двойные покупки

---

### 4. **Автоматическая деактивация при запросе конфига** ✅

**Где:** `bot.py` → `myconfig_command()`
```python
if not is_active:
    db.deactivate_user(user.id)  # Деактивирует при проверке
```

**Результат:** Автоматически помечает подписку как неактивную

---

### 5. **Функция получения истекших пользователей** ✅

**Где:** `database.py` → `get_expired_users()`
```python
SELECT user_id, username, expiry_date
FROM users 
WHERE expiry_date < ? AND is_active = 1
```

**Результат:** Можно найти всех пользователей с истекшей подпиской

## ⚠️ **ВАЖНОЕ ОГРАНИЧЕНИЕ для Shadowsocks:**

### ❌ **Нет серверной блокировки доступа**

**Проблема:**
- Все пользователи Shadowsocks используют **ОДИН пароль сервера**
- Если пользователь уже установил конфиг на устройство, он может продолжать использовать VPN даже после истечения подписки
- Shadowsocks сервер не проверяет подписки - он просто принимает соединения с правильным паролем

**В отличие от WireGuard:**
- WireGuard: Можно удалить peer с сервера → подключение прекращается сразу
- Shadowsocks: Нет механизма удаления конкретного пользователя с сервера

---

## 🔧 **Решения для полной блокировки:**

### **Вариант 1: Периодическая смена пароля сервера** (Рекомендуется)

**Как работает:**
1. Периодически (например, раз в месяц) меняем пароль сервера
2. Генерируем новые конфиги для активных пользователей
3. Пользователи с истекшей подпиской не получат новый конфиг
4. Старые конфиги перестанут работать

**Преимущества:**
- ✅ Полная блокировка доступа
- ✅ Работает автоматически
- ✅ Простая реализация

**Недостатки:**
- ⚠️ Все активные пользователи должны обновить конфиг

---

### **Вариант 2: Использование ACL (Access Control List)**

**Как работает:**
- Использовать Shadowsocks сервер с поддержкой ACL (например, через дополнительный прокси)
- Хранить список разрешенных user_id
- Проверять при каждом подключении

**Преимущества:**
- ✅ Точная блокировка по пользователю
- ✅ Не требует смены пароля

**Недостатки:**
- ⚠️ Сложнее в настройке
- ⚠️ Требует дополнительного ПО

---

### **Вариант 3: Ротация конфигов для каждого пользователя**

**Как работает:**
- Для каждого пользователя генерировать уникальный пароль
- При истечении подписки - удалять пароль из базы
- Shadowsocks сервер с поддержкой множественных паролей

**Преимущества:**
- ✅ Блокировка конкретного пользователя
- ✅ Без смены пароля для всех

**Недостатки:**
- ⚠️ Требует модификации Shadowsocks сервера
- ⚠️ Более сложная архитектура

---
